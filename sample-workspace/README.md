# Ant Workbench debug sample workspace

Pressing F5 launches the Extension Development Host with this folder opened automatically.

Every debug session starts clean: the `prepare-debug` preLaunchTask (see `.vscode/tasks.json` and
`scripts/clean-sample-workspace.js`) wipes any previously generated `.classpath`, `.project`,
`.vscode/settings.json`, and build output before launch. You never need to delete those by hand —
doing so risks taking the wrong thing with them (e.g. deleting `build.xml` while aiming for
`.classpath`).

## Layout

```text
sample-workspace/
├── MyApp/
│   ├── build/
│   │   ├── build.xml          main build file: clean / compile / run (default) / jar
│   │   └── build-app.xml      imports build.xml, adds a `package` target
│   ├── lib/
│   │   ├── sample-lib.jar     Slogan
│   │   └── test/
│   │       └── sample-test-lib.jar   TestSlogan
│   └── src/com/example/myapp/
│       ├── App.java           imports Slogan (external jar); calls Greeter (same package)
│       └── Greeter.java
└── MyApp-test/
    └── src/com/example/myapptest/
        └── GreeterTest.java   imports Greeter (cross-project) and TestSlogan (jar)
```

### `build.xml` / `build-app.xml`

Two build files in the same folder, so the sidebar tree lists both and Active Build File
switching has something to switch between. `build-app.xml` imports `build.xml` and adds one
`package` target (zips up `myapp.jar`).

### `MyApp/lib/sample-lib.jar` (`Slogan`)

The only import in this sample that actually depends on classpath resolution:

- `App.java` imports and calls it, so a broken/stale generated classpath would show up as a real
  compile or run failure — not just "look configured".
- `lib/` is nested inside `MyApp/` rather than sitting at the workspace root on purpose. VS
  Code's Language Support for Java auto-references a workspace-root `lib/**/*.jar` by default —
  if `lib/` were at `sample-workspace/lib`, `Slogan` would resolve even without a generated
  `.classpath`, defeating the point of this sample. Nested here, it stays genuinely dependent on
  **Generate .classpath**: until that runs, the import shows red squiggles in the editor.
- `Greeter`, by contrast, resolves regardless of `.classpath` — it's in the same project and
  package as `App.java`, so same-project source is always visible to both Ant and the editor.
  Only `Slogan` proves anything about classpath resolution.

### `MyApp-test/` (`GreeterTest.java`)

Deliberately _not_ pre-wired in `.vscode/settings.json`: `antWorkbench.additionalClasspaths` is
meant to be generated by using **Add Classpath Target** (see "Things to try" below), not
hand-authored ahead of time.

`GreeterTest.java` lives in its own package (`com.example.myapptest`, distinct from
`com.example.myapp`), so both of its imports are genuine cross-references rather than
same-package freebies:

- `Greeter` — from `MyApp`, resolved through a `projectDeps` source reference.
- `TestSlogan` — from `sample-test-lib.jar`, resolved through `test-classpath`.

Until `additionalClasspaths` is configured and Generate .classpath has run here, both imports show
red squiggles, and removing either one causes a real "cannot find symbol" compile error.

## Things to try

1. 🌳 Confirm the Ant Workbench sidebar tree shows both `build.xml` and `build-app.xml`.

2. ▶️ Run the `run` target on `build.xml` (requires Ant). The console prints:

   ```text
   Hello, Ant Workbench! (resolved via Ant Workbench classpath)
   ```

   This proves `Slogan` was resolved from `MyApp/lib/sample-lib.jar` via the compile classpath.

3. 🔗 Click a target name to jump to its `<target>` definition in the build file.

4. 📄 Run **Generate .classpath** on `build.xml` (requires Ant + Language Support for Java).
   Writes `MyApp/.classpath` and `MyApp/.project`.

5. ➕ Click **Add Classpath Target** on `build.xml`:

   - Pick `test-classpath` from the QuickPick.
   - Choose `MyApp-test` as the output directory.
   - Pick `MyApp` in the project-reference QuickPick that follows.

   This adds an entry (with `projectDeps: ["MyApp"]`) to `antWorkbench.additionalClasspaths` in
   `.vscode/settings.json` (created on first use).

   Run **Generate .classpath** again to write `MyApp-test/.classpath` and `.project`. Once
   generated, `GreeterTest.java`'s imports resolve (no red squiggles), and it can be run as a Java
   application to confirm both the `MyApp` project reference and `test-classpath` actually work —
   not just look configured.

6. ✅ Set `build-app.xml` as the active build file and confirm the active indicator (green check
   icon) moves.

7. 👋 To see the "no build files found" welcome message, temporarily rename `MyApp/build/` and
   reload.
